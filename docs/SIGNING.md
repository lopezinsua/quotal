# Code signing status & readiness

Single source of truth for Quotal's installer signing. Honest summary: **the
installers are not OS-code-signed yet** — that's gated on getting a certificate,
not on code. Everything in the repo is already wired so that signing turns on the
moment a certificate/account exists (just add secrets).

## What already protects downloads today (no certificate needed)

- **SHA-256 checksums.** Every release ships `SHA256SUMS.txt` (generated by the
  `checksums` job in `.github/workflows/release.yml`). Users can verify integrity
  before running — see the README's "Verify your download".
- **Signed auto-updates.** In-app updates are verified end-to-end with a
  [minisign](https://jedisct1.github.io/minisign/) key (public key in
  `tauri.conf.json → plugins.updater.pubkey`; signing via `TAURI_SIGNING_PRIVATE_KEY`
  in CI). A tampered or spoofed update is rejected. This is **not** OS code signing,
  but it does secure the update channel.

So the unsigned-installer warning (Windows SmartScreen / macOS Gatekeeper) is about
an *unknown publisher*, not tampering — and the two mechanisms above already let a
user confirm authenticity.

## Why not paid signing

A standard OV/EV certificate (Windows) or the Apple Developer Program (macOS, $99/yr)
is a **recurring cost**. As a free MIT project, Quotal pursues **free** routes instead.

## The plan, per platform

### Windows — SignPath Foundation (in progress)

- **Status:** application form **submitted**, awaiting approval.
- SignPath Foundation grants free code-signing certificates to eligible OSS.
- Once approved, wire the SignPath GitHub Action into `release.yml` to submit the
  built `.msi`/`.exe` for signing (SignPath signs server-side; no private key in CI).

There is also an **already-wired Authenticode path** in `release.yml` (the
"Configurar firma de Windows" step) that activates if a `WINDOWS_CERTIFICATE` (PFX,
base64) + `WINDOWS_CERTIFICATE_PASSWORD` secret pair is present. That covers a
classic PFX cert; SignPath's server-side flow is the preferred free route.

> Note: an OV cert still needs SmartScreen "reputation" to accumulate; EV grants it
> immediately but typically requires an HSM token (not exportable to a PFX, so it
> doesn't fit the PFX flow).

### macOS — Developer ID + notarization (not started)

- Needs an **Apple Developer Program** account ($99/yr) — no free route exists.
- The `release.yml` `tauri-action` step already lists the required env vars
  (`APPLE_CERTIFICATE`, `APPLE_CERTIFICATE_PASSWORD`, `APPLE_SIGNING_IDENTITY`,
  `APPLE_ID`, `APPLE_PASSWORD`, `APPLE_TEAM_ID`), **commented out**. Uncomment and add
  the secrets **only when an account exists**. Notarization runs automatically once
  `APPLE_ID` + `APPLE_PASSWORD` + `APPLE_TEAM_ID` are set.

### Linux

No OS signing model like Windows/macOS. The `SHA256SUMS.txt` + minisign-signed
updates are the integrity story.

## Readiness checklist (when a cert/account arrives)

- **Windows (SignPath):** add the SignPath submit-signing-request step to
  `release.yml` after the build, pointing at the SignPath org/project/policy; store
  the SignPath API token as a secret. Then drop the "unsigned" note from the README.
- **Windows (PFX, alternative):** add `WINDOWS_CERTIFICATE` (base64 PFX) +
  `WINDOWS_CERTIFICATE_PASSWORD` secrets — the existing step picks them up, no code
  change needed.
- **macOS:** add the six `APPLE_*` secrets and uncomment the block in `release.yml`.
- Update the README "unidentified developer" note once a platform is signed.

## Files involved

- `.github/workflows/release.yml` — Windows signing step (conditional), macOS env
  block (commented), and the `checksums` job.
- `src-tauri/tauri.conf.json` — updater `pubkey` (minisign).
